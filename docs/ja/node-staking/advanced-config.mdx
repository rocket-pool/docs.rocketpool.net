# DockerモードのSmartnode高度な設定

このセクションでは、Smartnodeのカスタマイズと機能を拡張するいくつかのテクニックを説明します。

## 外部バリデータークライアントをSmartnodeに接続できるようにする

SmartnodeはExecution ClientとBeacon Nodeを管理しているため、それらの設定（アップデートを含む）の多くを自動的に処理します。
ユーザーからよくあるリクエストの一つは、外部管理のバリデータークライアントをSmartnodeのExecution ClientおよびConsensus Clientに接続する機能です。

本質的に、これは一種の**「リバースハイブリッド」モード**として機能します。SmartnodeはExecution ClientとConsensus Clientを自前で管理し、外部のバリデータークライアントがそのクライアントを利用できます。

これを有効にするのは、[設定マネージャー](./config-docker#configuring-via-the-settings-manager)を使えば簡単です。

### Execution Client

Execution Clientへのアクセスを有効にするには、**Execution Client（ETH1）**カテゴリーに移動し、**Expose RPC Ports**ボックスにチェックを入れます：

<img src={tui_ec_expose_ports} width="100%" height="auto" />

これにより、ノードのRPCポート（HTTPのデフォルト8545およびWebSocketsのデフォルト8546）が開かれ、ローカルネットワーク上の他のマシンがそれらに接続できるようになります。

例えば、ノードのローカルIPアドレスが`192.168.1.45`の場合、このオプションを有効にすることで`http://192.168.1.45:8545`を他のサービスのExecution Client URLとして使用できます。

### Beacon Node（Consensus Client）

Consensus ClientのBeacon Nodeへのアクセスを有効にするには、**Consensus Client（ETH2）**カテゴリーに移動し、**Expose API Port**ボックスにチェックを入れます：

<img src={tui_cc_expose_ports} width="100%" height="auto" />

これにより、ノードのHTTP APIポート（デフォルト5052）が開かれ、ローカルネットワーク上の他のマシンがそれに接続できるようになります。

::: warning 注意：iptablesを介して開かれるポート（uwfではない）
RocketpoolコンフィギュレーターでRPCおよびAPIエンドポイントを公開すると、ポートは`iptables`を介して開かれるため、`ufw`ファイアウォールルールには表示されません。開いているポートを確認するには`sudo lsof -i -P -n | grep LISTEN`などのコマンドを使用できます。
:::

例えば、ノードのローカルIPアドレスが`192.168.1.45`の場合、このオプションを有効にすることで`http://192.168.1.45:5052`を他のサービスのBeacon Node URLとして使用できます。

::: warning PRYSMユーザーへの注意
**Prysm**には`Expose RPC Port`という2つ目のボックスがあり、外部管理のPrysm Validator ClientをBeacon Nodeに接続したい場合はチェックを入れる必要があります。
これはPrysmのValidator Clientが現在、標準のBeacon HTTP APIを介してBeacon Nodeに接続できないためです。
将来的にはこの問題が修正され、他のクライアントと同様にチェックボックスが1つになる予定です。
:::

### MEV-Boost

MEV-Boostへのアクセスを有効にするには、**MEV-Boost**カテゴリーに移動し、**Expose API Port**ボックスにチェックを入れます：

<img src={tui_mev_expose_port} width="100%" height="auto" />

これにより、ノードのHTTP APIポート（デフォルト18550）が開かれ、ローカルネットワーク上の他のマシンがそれに接続できるようになります。

例えば、ノードのローカルIPアドレスが`192.168.1.45`の場合、このオプションを有効にすることで`http://192.168.1.45:18550`を他のサービスのMEV-Boost URLとして使用できます。

## Docker-Compose定義ファイルのカスタマイズ

設定マネージャーにはノードをカスタマイズするための多くのオプションが含まれていますが、`docker-compose`が提供するすべての機能を網羅しているわけではありません。
一部のユーザーはSmartnodeの`docker-compose`ファイルに補足的なカスタマイズを必要とします。

`docker-compose`ファイルは`rocketpool service start`が呼び出されるたびにテンプレート（`templates`ディレクトリ内）からランタイムファイル（`runtime`ディレクトリ内）に再生成されるため、ランタイムファイルを直接変更することはお勧めしません。
代わりに、`docker-compose`の[組み込みオーバーライド機能](https://docs.docker.com/compose/extends/#adding-and-overriding-configuration)を活用して、再起動後も保持される必要なカスタマイズを行います。
さらに、このシステムはSmartnodeスタックのアップグレードまたは再インストール後も**変更を保持します**。

これを行うには、`override`フォルダー内の対象コンテナのファイルを開き、カスタマイズを追加するだけです。
これらのファイルに記述した内容は、`runtime`フォルダーの`docker-compose`ファイルにある対応する設定を**オーバーライド**します。

例えば、[CloudWatchサポートのためにGrafanaにAWS認証情報を追加](https://grafana.com/docs/grafana/latest/administration/configure-docker/#configure-aws-credentials-for-cloudwatch-support)したい場合、次のように`override/grafana.yml`を変更します：

```yaml
# Enter your own customizations for the Grafana container here. These changes will persist after upgrades, so you only need to do them once.
#
# See https://docs.docker.com/compose/extends/#adding-and-overriding-configuration
# for more information on overriding specific parameters of docker-compose files.

version: "3.7"
services:
  grafana:
    x-rp-comment: Add your customizations below this line
    environment:
      - GF_AWS_PROFILES=default
      - GF_AWS_default_ACCESS_KEY_ID=YOUR_ACCESS_KEY
      - GF_AWS_default_SECRET_ACCESS_KEY=YOUR_SECRET_KEY
      - GF_AWS_default_REGION=us-east-1
```

これにより、ここに記載された4つの環境変数がGrafanaコンテナの既存の環境変数配列にマージされます。

あるいは、eth1クライアントデータをファイルシステムのデフォルト以外のパスに保存したい場合、次のようなボリュームパラメーターを含むように`override/eth1.yml`を変更します：

```yaml
# Enter your own customizations for the eth1 container here. These changes will persist after upgrades, so you only need to do them once.
#
# See https://docs.docker.com/compose/extends/#adding-and-overriding-configuration
# for more information on overriding specific parameters of docker-compose files.

version: "3.7"
services:
  eth1:
    x-rp-comment: Add your customizations below this line
volumes:
  eth1clientdata:
    driver: local
    driver_opts:
      type: none
      o: bind
      device: "/data/eth/eth1"
```

これにより、パラメーターが`eth1clientdata`ボリューム仕様にマージされ、eth1クライアントデータの場所がデフォルト（Dockerのデフォルト設定では`/var/lib/docker/volumes/rocketpool_eth1clientdata`）ではなく`/data/eth/eth1`になります。

`docker-compose`オーバーライドの動作の詳細については、[https://docs.docker.com/compose/extends/#adding-and-overriding-configuration](https://docs.docker.com/compose/extends/#adding-and-overriding-configuration)をご覧ください。

## Node Daemon HTTP API

Node daemonは、ポート8280（localhostのみ）でローカルHTTP APIを公開します。このAPIはCLIによって内部的に使用され、カスタム自動化やツール用に直接呼び出すこともできます。

HTTP APIのポートは、Smart NodeとTX Feesカテゴリーの`API port`で調整できます。

<img src={tui_api_port} width="100%" height="auto" />

外部公開は現在サポートされていません。安全なリモートアクセスの最適な方法を検討している間、APIはlocalhostでのみ起動します。
現時点では、ポートを公開する設定オプションはありません。

標準のHTTPリクエストを使用してAPIを呼び出すことができます。

- 読み取り専用クエリ（GET）:
```shell
curl -s http://localhost:8280/api/node/sync | jq
```
```json
{
  "status": "success",
  "error": "",
  "ecStatus": {
    "primaryEcStatus": {
      "isWorking": true,
      "isSynced": true,
      "syncProgress": 1,
      "networkId": 560048
    }
  },
  "bcStatus": {
    "primaryEcStatus": {
      "isWorking": true,
      "isSynced": true,
      "syncProgress": 1
    }
  }
}
```

- トランザクション（POST）:
```shell
curl -s -X POST http://localhost:8280/api/node/set-timezone \
  -d "timezoneLocation=America/Los_Angeles" | jq
```
```json
{
  "status": "success",
  "error": "",
  "txHash": "0x0bab800962146f3e41c701601be6d636f9bec261ef6cabe667bc560ed19dfffb"
}
```

## （実験的）デュアルスタック（IPv4 + IPv6）ネットワークの有効化

このオプションは、ローカル管理クライアントでDockerモード設定を実行しているユーザー向けです。
Smart node v1.20.2以降、Rocket PoolはEthereumクライアントのデュアルスタック（IPv4 + IPv6）ネットワークをサポートしています。

有効にすると、コンセンサスクライアントはIPv4とIPv6の両方のピアに接続でき、ピア発見が改善される可能性があります。
この機能は、ネットワークがCGNATの背後にあり、IPv4でのピアリングに困難がある場合にも役立ちます。


::: tip ヒント
以下のサイトでEthereum IPv6サポートの現状を確認できます。

[ipv6eth.info](https://ipv6eth.info/)
:::

::: warning 実験的機能
IPv6サポートは現在実験的です。サポートされているすべてのコンセンサスクライアントでテストされていますが、ネットワーク設定によっては問題が発生する場合があります。
問題が発生した場合は、IPv6を無効にしてクライアントを再起動することでIPv4のみモードに戻ることができます。IPv6サポートで問題が発生した場合は、[Rocket Pool Discord](https://discord.gg/rocketpool)の**#support**チャンネルで報告してください。

:::

IPv6サポートを有効にした場合：

- DockerネットワークがデュアルスタックIPv4 + IPv6）用に設定されます
- コンセンサスクライアントがIPv4とIPv6の両方のアドレスでリッスンします
- ノードがIPv4ピアに加えてIPv6ピアを発見し接続できるようになります
- ノードがそのIPv6アドレスをENR（Ethereum Node Record）にアドバタイズし、他のIPv6ノードから発見可能になります

### クライアント固有の動作

コンセンサスクライアントごとにIPv6サポートの程度が異なります：

| クライアント | IPv6リッスン | IPv6ブロードキャスト（ENR） | ピアリング動作 | 備考 |
|--------|----------------|------------------------|------------------|-------|
| **Teku** | 完全 | 完全 | IPv6を優先 | 利用可能な場合はIPv6接続を優先。専用IPv6ポートを使用（デフォルト：9090）。 |
| **Lighthouse** | 完全 | 完全 | 優先なし | 先に応答したアドレス（IPv4またはIPv6）に接続。通常、IPv6ピアは約1〜2%。 |
| **Lodestar** | 完全 | 完全 | IPv4を優先 | IPv4とIPv6の両方でリッスンおよびブロードキャストするが、相手ノードがIPv4を無効にしている場合のみIPv6でピアリング。 |
| **Nimbus** | 完全 | IPv4のみ* | 主にIPv4 | インバウンドIPv6接続を受け入れられるが、ENRにはIPv4アドレスのみをブロードキャスト。 |
| **Prysm** | 完全 | IPv4のみ* | 主にIPv4 | インバウンドIPv6接続を受け入れられるが、ENRにはIPv4アドレスのみをブロードキャスト。 |

\* デフォルトではIPv4アドレスをブロードキャスト。IPv6のみのホストでは、代わりにIPv6をブロードキャスト。


### 前提条件

IPv6を有効にする前に、以下を確認してください：

1. ISP/ホストがIPv6接続を提供している - [ipv6test.google.com/](https://ipv6test.google.com/)でテストするか、CLIで次のコマンドを実行してください：`curl -6 https://icanhazip.com`。コマンドが`xxxx:xxxx:xxxx:xxxx:xxxx:xxxx:xxxx:xxxx`形式のIPv6アドレスを返した場合は準備完了です。
2. ファイアウォールがコンセンサスクライアントP2PポートでIPv6トラフィックを許可している。デフォルトP2Pポートは`9001`です。TekuにはIPv6用の追加専用ポートが必要です。デフォルトのTeku P2P IPv6ポートは`9090`です。
3. Dockerモードでローカル管理クライアントを使用している（外部管理クライアントではない）

### IPv6を有効にする方法


IPv6を有効にするには、Smart NodeとTX Feesカテゴリーに移動し、**Enable IPv6**ボックスにチェックを入れてから保存してSmart Nodeスタックを再起動します。
<img src={tui_enable_ipv6} width="100%" height="auto" />

::: warning 注意
IPv6を有効にしているがパブリックIPv6アドレスがない場合、Smart nodeは起動時に警告を表示します：
```
Warning: IPv6 is enabled but no public IPv6 address was detected.
Your node will not be reachable by IPv6 peers.
```
ノードはIPv4を使用して通常通り動作し続けます。
:::

### IPv6が機能していることの確認

IPv6を有効にしてサービスを再起動した後、正しく機能していることを確認できます：

1. Dockerネットワーク設定を確認する

```shell
docker inspect rocketpool_net | grep -i ipv6
```

`"EnableIPv6": true`と表示され、各コンテナに`IPv6Address`が割り当てられているはずです。

2. ポートバインディングを確認する

```shell
ss -tlnp | grep 9001
```

`[::]:9001`（IPv6ワイルドカードアドレス）が表示されるはずです。

3. パブリックIPv6アドレスを確認する

```shell
curl -6 https://icanhazip.com
```

パブリックIPv6アドレス（例：`xxx:xxx:xxxx:...`）が返されるはずです。

4. コンセンサスクライアントのログを確認する

コンセンサスクライアントのログを表示してIPv6リッスンを確認します：

```shell
docker logs rocketpool_eth2 2>&1 | grep -iE "listen|ipv6|::" | head -20
```

**確認すべき内容：**

- **Lighthouse** - `Listening on /ip6/`または`listen_addr: /ip6/`
- **Lodestar** - `/ip4/`と`/ip6/`の両アドレスを示す`Multiaddrs`
- **Teku** - `0.0.0.0`と`::`の両方を示す`p2p-interface`
- **Nimbus** - `Listening to incoming network requests`（IPv6について明示的に言及しない）
- **Prysm** - `Beacon chain gRPC server listening`（IPv6について明示的に言及しない）

5. IPv6ピアを確認する

APIを介してビーコンノードのピアリストを確認します：

```shell
curl -s http://localhost:5052/eth/v1/node/peers | grep -o '/ip6/[^"]*' | head -10
```

接続されているIPv6ピアアドレスが最大10件表示されます。使用しているコンセンサスクライアントによっては、IPv6ピアが接続されるまでしばらく時間がかかる場合があります。テストでは、TekuがIPv6ピアに即座に接続した唯一のクライアントであることがわかりました。
