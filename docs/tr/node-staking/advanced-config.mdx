# Docker Modu için Gelişmiş Smartnode Yapılandırması

Bu bölümde, Smartnode'un özelleştirme ve işlevselliğini genişleten birkaç tekniği ele alacağız.

## Harici Doğrulayıcı İstemcilerinin Smartnode'a Bağlanmasına İzin Verme

Smartnode, Yürütme istemcisini ve Beacon Node'u yönettiğinden, yapılandırmalarının (güncellemeler dahil) büyük bölümünü otomatik olarak halleder.
Kullanıcıların sık dile getirdiği bir istek, harici olarak yönetilen bir Doğrulayıcı istemcisini Smartnode'un Yürütme ve Konsensüs istemcilerine bağlayabilme yeteneğidir.

Bu, özünde bir tür **"Ters Hibrit" mod** olarak işlev görür; Smartnode kendi Doğrulayıcı istemcisini yönetip harici bir Yürütme ve Konsensüs istemcisine bağlanmak yerine, üçünü birden yönetir ve harici bir Doğrulayıcı istemcisi kendi istemcilerini kullanabilir.

Bunu etkinleştirmek, [Ayarlar Yöneticisi](./config-docker#configuring-via-the-settings-manager)'ni kullanarak son derece basittir.

### Yürütme İstemcisi

Yürütme istemcisine erişimi etkinleştirmek için **Yürütme İstemcisi (ETH1)** kategorisine gidin ve **RPC Portlarını Aç** kutusunu işaretleyin:

<img src={tui_ec_expose_ports} width="100%" height="auto" />

Bu işlem, RPC portlarını (HTTP için varsayılan 8545 ve WebSocket için 8546) düğümünüzde açarak yerel ağınızdaki diğer makinelerin bunlara bağlanabilmesini sağlar.

Örneğin, düğümünüzün yerel IP adresi `192.168.1.45` ise, bu seçeneği etkinleştirerek diğer hizmetleriniz için Yürütme istemcisi URL'si olarak `http://192.168.1.45:8545` kullanabilirsiniz.

### Beacon Node (Konsensüs İstemcisi)

Konsensüs istemcisinin Beacon Node'una erişimi etkinleştirmek için **Konsensüs İstemcisi (ETH2)** kategorisine gidin ve **API Portunu Aç** kutusunu işaretleyin:

<img src={tui_cc_expose_ports} width="100%" height="auto" />

Bu işlem, HTTP API portunu (varsayılan 5052) düğümünüzde açarak yerel ağınızdaki diğer makinelerin buna bağlanabilmesini sağlar.

::: warning Not: Portlar iptables aracılığıyla açılır, ufw ile değil
Rocketpool yapılandırıcısını kullanarak RPC ve API uç noktalarını açtığınızda, portlar `iptables` aracılığıyla açılır ve bu nedenle `ufw` güvenlik duvarı kuralı olarak görünmez. Hangi portların açık olduğunu görmek için `sudo lsof -i -P -n | grep LISTEN` veya benzer bir komut kullanabilirsiniz.
:::

Örneğin, düğümünüzün yerel IP adresi `192.168.1.45` ise, bu seçeneği etkinleştirerek diğer hizmetleriniz için Beacon Node URL'si olarak `http://192.168.1.45:5052` kullanabilirsiniz.

::: warning PRYSM KULLANICILARI İÇİN NOT
**Prysm**'in, harici olarak yönetilen bir Prysm Doğrulayıcı İstemcisini Beacon Node'unuza bağlamak istiyorsanız işaretlenmesi gereken `RPC Portunu Aç` adlı ikinci bir kutusu vardır.
Bunun nedeni, Prysm'in Doğrulayıcı İstemcisinin şu anda standart Beacon HTTP API aracılığıyla bir Beacon Node'a bağlanamamasıdır.
Gelecekte bu düzeltilecek ve diğer istemciler gibi yalnızca bir onay kutusu olacaktır.
:::

### MEV-Boost

MEV-Boost'a erişimi etkinleştirmek için **MEV-Boost** kategorisine gidin ve **API Portunu Aç** kutusunu işaretleyin:

<img src={tui_mev_expose_port} width="100%" height="auto" />

Bu işlem, HTTP API portunu (varsayılan 18550) düğümünüzde açarak yerel ağınızdaki diğer makinelerin buna bağlanabilmesini sağlar.

Örneğin, düğümünüzün yerel IP adresi `192.168.1.45` ise, bu seçeneği etkinleştirerek diğer hizmetleriniz için MEV-Boost URL'si olarak `http://192.168.1.45:18550` kullanabilirsiniz.

## Docker-Compose Tanım Dosyalarını Özelleştirme

Ayarlar Yöneticisi, düğümünüzü özelleştirmek için kullanabileceğiniz birçok seçenek içerse de `docker-compose`'un sağladığı tüm özellikleri kapsamaz.
Bazı kullanıcılar, Smartnode'un `docker-compose` dosyalarına ek özelleştirmeler yapmayı gerektirir.

`docker-compose` dosyaları, `rocketpool service start` çağrıldığında her seferinde şablonlardan (`templates` dizini) yeniden oluşturulduğundan (`runtime` dizinine), çalışma zamanı dosyalarını değiştirmek önerilmez.
Bunun yerine, `docker-compose`'un [yerleşik geçersiz kılma işlevselliğinden](https://docs.docker.com/compose/extends/#adding-and-overriding-configuration) yararlanarak yeniden başlatmalarda korunan gerekli değişiklikleri yapabilirsiniz.
Üstelik bu sistem, **Smartnode yığınının yükseltmesinden veya yeniden kurulumundan sonra da değişikliklerinizi korur**.

Bunu yapmak için, `override` klasöründe tercih ettiğiniz konteynere ait dosyayı açın ve özelleştirmelerinizi buraya ekleyin.
Bu dosyalara koyduğunuz her şey, `runtime` klasöründeki `docker-compose` dosyalarındaki ilgili ayarları **geçersiz kılacaktır**.

Örneğin, [CloudWatch desteği için Grafana'ya AWS kimlik bilgileri eklemek](https://grafana.com/docs/grafana/latest/administration/configure-docker/#configure-aws-credentials-for-cloudwatch-support) istediğinizi varsayalım.
Bunu `override/grafana.yml` dosyasını şu şekilde değiştirerek yapabilirsiniz:

```yaml
# Enter your own customizations for the Grafana container here. These changes will persist after upgrades, so you only need to do them once.
#
# See https://docs.docker.com/compose/extends/#adding-and-overriding-configuration
# for more information on overriding specific parameters of docker-compose files.

version: "3.7"
services:
  grafana:
    x-rp-comment: Add your customizations below this line
    environment:
      - GF_AWS_PROFILES=default
      - GF_AWS_default_ACCESS_KEY_ID=YOUR_ACCESS_KEY
      - GF_AWS_default_SECRET_ACCESS_KEY=YOUR_SECRET_KEY
      - GF_AWS_default_REGION=us-east-1
```

Bu işlem, burada listelenen dört ortam değişkenini Grafana konteynerindeki mevcut ortam değişkenleri dizisiyle birleştirecektir.

Alternatif olarak, eth1 istemci verilerini dosya sisteminizde varsayılan olmayan bir yola kaydetmek istiyorsanız, `override/eth1.yml` dosyasını şu gibi birim parametreleri içerecek şekilde değiştirirsiniz:

```yaml
# Enter your own customizations for the eth1 container here. These changes will persist after upgrades, so you only need to do them once.
#
# See https://docs.docker.com/compose/extends/#adding-and-overriding-configuration
# for more information on overriding specific parameters of docker-compose files.

version: "3.7"
services:
  eth1:
    x-rp-comment: Add your customizations below this line
volumes:
  eth1clientdata:
    driver: local
    driver_opts:
      type: none
      o: bind
      device: "/data/eth/eth1"
```

Bu işlem, parametreleri `eth1clientdata` birim spesifikasyonuyla birleştirerek `/data/eth/eth1`'i eth1 istemci verilerinin konumu olarak belirler; varsayılan konum yerine (`/var/lib/docker/volumes/rocketpool_eth1clientdata` olur — varsayılan Docker yapılandırması kullanılıyorsa).

`docker-compose` geçersiz kılmalarının nasıl çalıştığı hakkında daha fazla bilgi için lütfen [https://docs.docker.com/compose/extends/#adding-and-overriding-configuration](https://docs.docker.com/compose/extends/#adding-and-overriding-configuration) adresini ziyaret edin.

## Node Daemon HTTP API

Node daemon, 8280 portunda (yalnızca localhost) yerel bir HTTP API sunar. Bu API, CLI tarafından dahili olarak kullanılır ve özel otomasyon veya araçlar için doğrudan da çağrılabilir.

HTTP API portu, Smart Node and TX Fees kategorisinde `API port` altından değiştirilebilir

<img src={tui_api_port} width="100%" height="auto" />

Harici olarak açma şu anda desteklenmemektedir. Güvenli uzaktan erişim için en iyi yaklaşımı belirlerken API yalnızca localhost üzerinde başlatılır.
Şu anda portu açmak için bir yapılandırma seçeneği yoktur.

API'yi standart HTTP istekleri kullanarak çağırabilirsiniz.

- Salt okunur sorgu (GET):
```shell
curl -s http://localhost:8280/api/node/sync | jq
```
```json
{
  "status": "success",
  "error": "",
  "ecStatus": {
    "primaryEcStatus": {
      "isWorking": true,
      "isSynced": true,
      "syncProgress": 1,
      "networkId": 560048
    }
  },
  "bcStatus": {
    "primaryEcStatus": {
      "isWorking": true,
      "isSynced": true,
      "syncProgress": 1
    }
  }
}
```

- İşlem (POST):
```shell
curl -s -X POST http://localhost:8280/api/node/set-timezone \
  -d "timezoneLocation=America/Los_Angeles" | jq
```
```json
{
  "status": "success",
  "error": "",
  "txHash": "0x0bab800962146f3e41c701601be6d636f9bec261ef6cabe667bc560ed19dfffb"
}
```

## (Deneysel) Çift Yığın (IPv4 + IPv6) Ağını Etkinleştirme

Bu seçenek, yerel olarak yönetilen istemcilerle Docker modu yapılandırmalarını kullanan kullanıcılar içindir.
Smart node v1.20.2'den itibaren Rocket Pool, Ethereum istemcileriniz için çift yığın (IPv4 + IPv6) ağını desteklemektedir.

Etkinleştirildiğinde, konsensüs istemciniz hem IPv4 hem de IPv6 eşleriyle bağlantı kurabilir; bu da potansiyel olarak eş keşfini iyileştirir.
Bu özellik aynı zamanda ağınız CGNAT arkasında olup IPv4 kullanarak eşleşmekte güçlük çekiyorsa da kullanışlıdır.


::: tip İPUCU
Aşağıdaki site, Ethereum IPv6 Desteği'nin durumuna ilişkin bir özet sunmaktadır.

[ipv6eth.info](https://ipv6eth.info/)
:::

::: warning DENEYSELÖZELLİK
IPv6 desteği şu anda deneysel aşamadadır. Desteklenen tüm konsensüs istemcilerinde test edilmiş olsa da ağ yapılandırmanıza bağlı olarak sorunlarla karşılaşabilirsiniz.
Sorun yaşarsanız IPv6'yı devre dışı bırakabilir ve yalnızca IPv4 moduna dönmek için istemcilerinizi yeniden başlatabilirsiniz. IPv6 desteğiyle ilgili herhangi bir sorunla karşılaşırsanız,
lütfen bunları [Rocket Pool Discord](https://discord.gg/rocketpool) sunucusundaki **#support** kanalında bildirin.

:::

IPv6 desteğini etkinleştirdiğinizde:

- Docker ağınız çift yığın (IPv4 + IPv6) için yapılandırılacaktır
- Konsensüs istemciniz hem IPv4 hem de IPv6 adreslerini dinleyecektir
- Düğümünüz IPv4 eşlerine ek olarak IPv6 eşlerini keşfedip bağlanabilecektir
- Düğümünüz IPv6 adresini ENR (Ethereum Düğüm Kaydı) içinde yayınlayacak, böylece diğer IPv6 düğümleri tarafından keşfedilebilir hale gelecektir

### İstemciye Özgü Davranış

Farklı konsensüs istemcileri IPv6'yı farklı destek düzeylerinde ele alır:

| İstemci | IPv6 Dinleme | IPv6 Yayını (ENR) | Eşleşme Davranışı | Notlar |
|---------|--------------|-------------------|-------------------|--------|
| **Teku** | Tam | Tam | IPv6'ya Öncelik Verir | Mevcut olduğunda IPv6 bağlantılarını tercih eder. Ayrı bir IPv6 portu kullanır (varsayılan: 9090). |
| **Lighthouse** | Tam | Tam | Tercih Yok | İlk yanıt veren adrese (IPv4 veya IPv6) bağlanır. Tipik olarak ~%1-2 IPv6 eşi görür. |
| **Lodestar** | Tam | Tam | IPv4'e Öncelik Verir | Hem IPv4 hem de IPv6'yı dinler ve yayınlar, ancak yalnızca diğer düğümün IPv4'ü devre dışıysa IPv6 üzerinden eşleşir. |
| **Nimbus** | Tam | Yalnızca IPv4* | Ağırlıklı IPv4 | Gelen IPv6 bağlantılarını kabul edebilir, ancak ENR'de yalnızca IPv4 yayınlar. |
| **Prysm** | Tam | Yalnızca IPv4* | Ağırlıklı IPv4 | Gelen IPv6 bağlantılarını kabul edebilir, ancak ENR'de yalnızca IPv4 yayınlar. |

\* Varsayılan olarak IPv4 adresini yayınlar. Yalnızca IPv6'ya sahip konakçılarda IPv6 yayınlayacaktır.


### Ön Koşullar

IPv6'yı etkinleştirmeden önce şunları sağlayın:

1. ISP/konakçınız IPv6 bağlantısı sağlamaktadır - Bunu [ipv6test.google.com/](https://ipv6test.google.com/) ile test edin
veya CLI'nızda şu komutu çalıştırın: `curl -6 https://icanhazip.com`. Komut `xxxx:xxxx:xxxx:xxxx:xxxx:xxxx:xxxx:xxxx` formatında bir IPv6 adresi döndürürse hazırsınız demektir.
2. Güvenlik duvarınız konsensüs istemcisi P2P portundan IPv6 trafiğine izin vermektedir. Varsayılan P2P portu `9001`'dir. Teku, IPv6 için ek bir özel port gerektirir. Varsayılan Teku P2P IPv6 portu `9090`'dır.
3. Docker modunda yerel olarak yönetilen istemciler kullanıyorsunuz (harici olarak yönetilen istemciler değil)

### IPv6 Nasıl Etkinleştirilir


IPv6'yı etkinleştirmek için Smart Node ve TX Ücretleri kategorisine gidin
ve **IPv6'yı Etkinleştir** kutusunu işaretleyin, ardından kaydedin ve Smart Node yığınını yeniden başlatın.
<img src={tui_enable_ipv6} width="100%" height="auto" />

::: warning NOT
IPv6'yı etkinleştirirseniz ancak genel bir IPv6 adresiniz yoksa, Smart node başlatma sırasında bir uyarı görüntüler:
```
Warning: IPv6 is enabled but no public IPv6 address was detected.
Your node will not be reachable by IPv6 peers.
```
Düğümünüz IPv4 kullanarak normal şekilde çalışmaya devam edecektir.
:::

### IPv6'nın Çalıştığını Doğrulama

IPv6'yı etkinleştirip hizmetlerinizi yeniden başlattıktan sonra, düzgün çalıştığını doğrulayabilirsiniz:

1. Docker Ağ Yapılandırmasını Kontrol Edin

```shell
docker inspect rocketpool_net | grep -i ipv6
```

`"EnableIPv6": true` görmelisiniz ve her konteynerin bir `IPv6Address`'i olmalıdır.

2. Port Bağlamasını Doğrulayın

```shell
ss -tlnp | grep 9001
```

`[::]:9001` (IPv6 joker adresi) görmelisiniz.

3. Genel IPv6 Adresinizi Onaylayın

```shell
curl -6 https://icanhazip.com
```

Genel IPv6 adresinizi döndürmelidir (ör. `xxx:xxx:xxxx:...`).

4. Konsensüs İstemcisi Günlüklerini Kontrol Edin

IPv6 dinlemeyi onaylamak için konsensüs istemcisi günlüklerinizi görüntüleyin:

```shell
docker logs rocketpool_eth2 2>&1 | grep -iE "listen|ipv6|::" | head -20
```

**Nelere dikkat etmeli:**

- **Lighthouse** - `Listening on /ip6/` veya `listen_addr: /ip6/`
- **Lodestar** - Hem `/ip4/` hem de `/ip6/` adreslerini gösteren `Multiaddrs`
- **Teku** - Hem `0.0.0.0` hem de `::` gösteren `p2p-interface`
- **Nimbus** - `Listening to incoming network requests` (IPv6'dan açıkça bahsetmez)
- **Prysm** - `Beacon chain gRPC server listening` (IPv6'dan açıkça bahsetmez)

5. IPv6 Eşlerini Kontrol Edin

API aracılığıyla beacon node'unuzun eş listesini kontrol edin:

```shell
curl -s http://localhost:5052/eth/v1/node/peers | grep -o '/ip6/[^"]*' | head -10
```

Bu, herhangi biri bağlıysa en fazla 10 IPv6 eş adresi gösterecektir. Kullandığınız konsensüs istemcisine bağlı olarak IPv6 eşlerinin bağlanması biraz zaman alabilir. Testlerimizde, IPv6 eşlerine anında bağlanan tek istemcinin Teku olduğunu gördük.
