# Расширенная настройка Smartnode для режима Docker

В этом разделе мы рассмотрим несколько техник, которые расширяют возможности настройки и функциональность Smartnode.

## Разрешение внешним клиентам валидатора подключаться к Smartnode

Поскольку Smartnode управляет клиентом исполнения и узлом Beacon, он автоматически обрабатывает большую часть их конфигурации (включая обновления).
Одним из распространённых запросов пользователей является возможность подключить внешний клиент валидатора к клиентам исполнения и консенсуса Smartnode.

По сути, это работает как своеобразный **«Обратный гибридный» режим**: вместо того чтобы Smartnode управлял собственным клиентом валидатора и подключался к внешним клиентам исполнения и консенсуса, он управляет всеми тремя компонентами, а внешний клиент валидатора может использовать его клиенты.

Включить это можно легко с помощью [Менеджера настроек](./config-docker#configuring-via-the-settings-manager).

### Клиент исполнения

Чтобы включить доступ к клиенту исполнения, перейдите в категорию **Execution Client (ETH1)** и установите флажок **Expose RPC Ports**:

<img src={tui_ec_expose_ports} width="100%" height="auto" />

Это откроет RPC-порты (по умолчанию 8545 для HTTP и 8546 для Websockets) на вашей ноде, чтобы другие машины в вашей локальной сети могли к ним подключаться.

Например, если локальный IP-адрес вашей ноды — `192.168.1.45`, то, включив этот параметр, вы сможете использовать `http://192.168.1.45:8545` в качестве URL клиента исполнения для других сервисов.

### Узел Beacon (Клиент консенсуса)

Чтобы включить доступ к узлу Beacon клиента консенсуса, перейдите в категорию **Consensus Client (ETH2)** и установите флажок **Expose API Port**:

<img src={tui_cc_expose_ports} width="100%" height="auto" />

Это откроет HTTP API-порт (по умолчанию 5052) на вашей ноде, чтобы другие машины в вашей локальной сети могли к нему подключаться.

::: warning Примечание: Порты открываются через iptables, а не ufw
Когда вы открываете RPC и API-эндпоинты с помощью конфигуратора Rocketpool, порты открываются через `iptables` и поэтому не будут отображаться как правило брандмауэра `ufw`. Чтобы увидеть, какие порты открыты, вы можете использовать команду `sudo lsof -i -P -n | grep LISTEN` или аналогичную.
:::

Например, если локальный IP-адрес вашей ноды — `192.168.1.45`, то, включив этот параметр, вы сможете использовать `http://192.168.1.45:5052` в качестве URL узла Beacon для других сервисов.

::: warning ПРИМЕЧАНИЕ ДЛЯ ПОЛЬЗОВАТЕЛЕЙ PRYSM
У **Prysm** есть второй флажок `Expose RPC Port`, который нужно отметить, если вы хотите подключить внешний клиент валидатора Prysm к вашему узлу Beacon.
Это связано с тем, что клиент валидатора Prysm в настоящее время не может подключаться к узлу Beacon через стандартный Beacon HTTP API.
В будущем это будет исправлено, и останется только один флажок, как у других клиентов.
:::

### MEV-Boost

Чтобы включить доступ к MEV-Boost, перейдите в категорию **MEV-Boost** и установите флажок **Expose API Port**:

<img src={tui_mev_expose_port} width="100%" height="auto" />

Это откроет HTTP API-порт (по умолчанию 18550) на вашей ноде, чтобы другие машины в вашей локальной сети могли к нему подключаться.

Например, если локальный IP-адрес вашей ноды — `192.168.1.45`, то, включив этот параметр, вы сможете использовать `http://192.168.1.45:18550` в качестве URL MEV-Boost для других сервисов.

## Настройка файлов определения Docker-Compose

Хотя Менеджер настроек содержит множество параметров для настройки вашей ноды, он не охватывает все функции `docker-compose`.
Некоторым пользователям требуются дополнительные настройки файлов `docker-compose` Smartnode.

Поскольку файлы `docker-compose` перегенерируются (в директорию `runtime`) из своих шаблонов (в директории `templates`) каждый раз при вызове `rocketpool service start`, изменять файлы среды выполнения не рекомендуется.
Вместо этого мы используем [встроенную функциональность переопределения `docker-compose`](https://docs.docker.com/compose/extends/#adding-and-overriding-configuration), чтобы вы могли вносить любые необходимые изменения, которые сохранятся после перезапуска.
Кроме того, эта система _также_ **сохраняет ваши изменения после обновления или переустановки стека Smartnode**.

Для этого просто откройте файл нужного контейнера в папке `override` и добавьте туда свои настройки.
Всё, что вы помещаете в эти файлы, будет **переопределять** соответствующие параметры в файлах `docker-compose` из папки `runtime`.

Например, предположим, вы хотите [добавить учётные данные AWS в Grafana для поддержки CloudWatch](https://grafana.com/docs/grafana/latest/administration/configure-docker/#configure-aws-credentials-for-cloudwatch-support).
Вы можете сделать это, изменив `override/grafana.yml` следующим образом:

```yaml
# Enter your own customizations for the Grafana container here. These changes will persist after upgrades, so you only need to do them once.
#
# See https://docs.docker.com/compose/extends/#adding-and-overriding-configuration
# for more information on overriding specific parameters of docker-compose files.

version: "3.7"
services:
  grafana:
    x-rp-comment: Add your customizations below this line
    environment:
      - GF_AWS_PROFILES=default
      - GF_AWS_default_ACCESS_KEY_ID=YOUR_ACCESS_KEY
      - GF_AWS_default_SECRET_ACCESS_KEY=YOUR_SECRET_KEY
      - GF_AWS_default_REGION=us-east-1
```

Это объединит четыре переменные среды, перечисленные здесь, с существующим массивом переменных среды в контейнере Grafana.

В качестве альтернативы, если вы хотите сохранить данные клиента eth1 по нестандартному пути в вашей файловой системе, вы можете изменить `override/eth1.yml` для добавления параметров тома:

```yaml
# Enter your own customizations for the eth1 container here. These changes will persist after upgrades, so you only need to do them once.
#
# See https://docs.docker.com/compose/extends/#adding-and-overriding-configuration
# for more information on overriding specific parameters of docker-compose files.

version: "3.7"
services:
  eth1:
    x-rp-comment: Add your customizations below this line
volumes:
  eth1clientdata:
    driver: local
    driver_opts:
      type: none
      o: bind
      device: "/data/eth/eth1"
```

Это объединит параметры в спецификацию тома `eth1clientdata`, сделав `/data/eth/eth1` местом хранения данных клиента eth1 вместо значения по умолчанию (`/var/lib/docker/volumes/rocketpool_eth1clientdata` при использовании конфигурации Docker по умолчанию).

Для получения дополнительной информации о том, как работают переопределения `docker-compose`, посетите [https://docs.docker.com/compose/extends/#adding-and-overriding-configuration](https://docs.docker.com/compose/extends/#adding-and-overriding-configuration).

## HTTP API демона ноды

Демон ноды предоставляет локальный HTTP API на порту 8280 (только localhost). Этот API используется CLI внутренне, а также может вызываться напрямую для пользовательской автоматизации или инструментов.

Порт HTTP API можно изменить в категории Smart Node and TX Fees в параметре `API port`

<img src={tui_api_port} width="100%" height="auto" />

Внешний доступ в настоящее время не поддерживается. API запускается только на localhost, пока мы определяем наилучший подход к безопасному удалённому доступу.
На данный момент нет опции конфигурации для открытия этого порта.

Вы можете вызывать API с помощью стандартных HTTP-запросов.

- Запрос только для чтения (GET):
```shell
curl -s http://localhost:8280/api/node/sync | jq
```
```json
{
  "status": "success",
  "error": "",
  "ecStatus": {
    "primaryEcStatus": {
      "isWorking": true,
      "isSynced": true,
      "syncProgress": 1,
      "networkId": 560048
    }
  },
  "bcStatus": {
    "primaryEcStatus": {
      "isWorking": true,
      "isSynced": true,
      "syncProgress": 1
    }
  }
}
```

- Транзакция (POST):
```shell
curl -s -X POST http://localhost:8280/api/node/set-timezone \
  -d "timezoneLocation=America/Los_Angeles" | jq
```
```json
{
  "status": "success",
  "error": "",
  "txHash": "0x0bab800962146f3e41c701601be6d636f9bec261ef6cabe667bc560ed19dfffb"
}
```

## (Экспериментально) Включение двойного стека (IPv4 + IPv6)

Этот параметр предназначен для пользователей, работающих с конфигурациями режима Docker с локально управляемыми клиентами.
Начиная со Smart node v1.20.2, Rocket Pool поддерживает двойной стек (IPv4 + IPv6) для ваших клиентов Ethereum.

При включении ваш клиент консенсуса сможет подключаться как к одноранговым узлам IPv4, так и IPv6, что потенциально улучшает обнаружение узлов.
Эта функция также полезна, если ваша сеть находится за CGNAT и испытывает трудности с подключением через IPv4.


::: tip СОВЕТ
На следующем сайте представлен обзор состояния поддержки IPv6 в Ethereum.

[ipv6eth.info](https://ipv6eth.info/)
:::

::: warning ЭКСПЕРИМЕНТАЛЬНАЯ ФУНКЦИЯ
Поддержка IPv6 в настоящее время является экспериментальной. Хотя она была протестирована со всеми поддерживаемыми клиентами консенсуса, вы можете столкнуться с проблемами в зависимости от конфигурации вашей сети.
Если у вас возникнут проблемы, вы можете отключить IPv6 и перезапустить клиентов, чтобы вернуться в режим только IPv4. Если вы столкнётесь с какими-либо проблемами с поддержкой IPv6,
пожалуйста, сообщите о них в канале **#support** на [Discord Rocket Pool](https://discord.gg/rocketpool).

:::

При включении поддержки IPv6:

- Ваша сеть Docker будет настроена для двойного стека (IPv4 + IPv6)
- Ваш клиент консенсуса будет слушать как на адресах IPv4, так и IPv6
- Ваша нода сможет обнаруживать и подключаться к одноранговым узлам IPv6 в дополнение к узлам IPv4
- Ваша нода будет объявлять свой адрес IPv6 в ENR (Ethereum Node Record), делая её обнаруживаемой другими узлами IPv6

### Поведение отдельных клиентов

Различные клиенты консенсуса обрабатывают IPv6 с разными уровнями поддержки:

| Клиент | Прослушивание IPv6 | Вещание IPv6 (ENR) | Поведение при подключении | Примечания |
|--------|-------------------|-------------------|--------------------------|------------|
| **Teku** | Полное | Полное | Приоритет IPv6 | Предпочитает соединения IPv6 при наличии. Использует отдельный порт IPv6 (по умолчанию: 9090). |
| **Lighthouse** | Полное | Полное | Без предпочтений | Подключается к тому адресу (IPv4 или IPv6), который отвечает первым. Обычно около 1-2% узлов IPv6. |
| **Lodestar** | Полное | Полное | Приоритет IPv4 | Слушает и вещает как на IPv4, так и на IPv6, но подключается через IPv6 только если у другого узла отключён IPv4. |
| **Nimbus** | Полное | Только IPv4* | В основном IPv4 | Может принимать входящие соединения IPv6, но вещает только IPv4 в ENR. |
| **Prysm** | Полное | Только IPv4* | В основном IPv4 | Может принимать входящие соединения IPv6, но вещает только IPv4 в ENR. |

\* По умолчанию вещает адрес IPv4. На хостах только с IPv6 вещает IPv6.


### Предварительные требования

Перед включением IPv6 убедитесь, что:

1. Ваш провайдер/хост обеспечивает подключение IPv6 — проверьте это на [ipv6test.google.com/](https://ipv6test.google.com/)
или выполнив эту команду в CLI: `curl -6 https://icanhazip.com`. Если команда возвращает адрес IPv6 в этом формате: `xxxx:xxxx:xxxx:xxxx:xxxx:xxxx:xxxx:xxxx`, вы готовы к работе.
2. Ваш брандмауэр разрешает трафик IPv6 на P2P-порту клиента консенсуса. По умолчанию P2P-порт — `9001`. Teku требует дополнительного выделенного
порта для IPv6. По умолчанию P2P-порт IPv6 Teku — `9090`.
3. Вы используете локально управляемые клиенты в режиме Docker (не внешне управляемые клиенты)

### Как включить IPv6


Чтобы включить IPv6, перейдите в категорию Smart Node and TX Fees
и установите флажок **Enable IPv6**, затем сохраните изменения и перезапустите стек Smart Node.
<img src={tui_enable_ipv6} width="100%" height="auto" />

::: warning ПРИМЕЧАНИЕ
Если вы включите IPv6, но у вас нет публичного адреса IPv6, Smart node отобразит предупреждение при запуске:
```
Warning: IPv6 is enabled but no public IPv6 address was detected.
Your node will not be reachable by IPv6 peers.
```
Ваша нода продолжит нормально работать через IPv4.
:::

### Проверка работы IPv6

После включения IPv6 и перезапуска сервисов вы можете убедиться, что всё работает корректно:

1. Проверьте конфигурацию сети Docker

```shell
docker inspect rocketpool_net | grep -i ipv6
```

Вы должны увидеть `"EnableIPv6": true`, и каждый контейнер должен иметь `IPv6Address`.

2. Проверьте привязку портов

```shell
ss -tlnp | grep 9001
```

Вы должны увидеть `[::]:9001` (wildcard-адрес IPv6).

3. Подтвердите ваш публичный адрес IPv6

```shell
curl -6 https://icanhazip.com
```

Должен вернуть ваш публичный адрес IPv6 (например, `xxx:xxx:xxxx:...`).

4. Проверьте логи клиента консенсуса

Просмотрите логи вашего клиента консенсуса для подтверждения прослушивания IPv6:

```shell
docker logs rocketpool_eth2 2>&1 | grep -iE "listen|ipv6|::" | head -20
```

**На что обратить внимание:**

- **Lighthouse** — `Listening on /ip6/` или `listen_addr: /ip6/`
- **Lodestar** — `Multiaddrs` с адресами `/ip4/` и `/ip6/`
- **Teku** — `p2p-interface` с `0.0.0.0` и `::`
- **Nimbus** — `Listening to incoming network requests` (явно не упоминает IPv6)
- **Prysm** — `Beacon chain gRPC server listening` (явно не упоминает IPv6)

5. Проверьте наличие узлов IPv6

Проверьте список одноранговых узлов вашего beacon node через API:

```shell
curl -s http://localhost:5052/eth/v1/node/peers | grep -o '/ip6/[^"]*' | head -10
```

Это отобразит до 10 адресов одноранговых узлов IPv6, если они подключены. В зависимости от используемого клиента консенсуса
для подключения узлов IPv6 может потребоваться некоторое время. В ходе нашего тестирования мы обнаружили, что Teku был единственным клиентом, немедленно подключающимся к узлам IPv6.
